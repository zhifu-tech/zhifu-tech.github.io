Hexo blog 部署指南

发表于 2025-10-09 分类于 技术 ， 工具
本项目支持三种部署方式，满足不同场景的需求:本地、远程和github pages

📋 概述

本项目支持三种部署方式，满足不同场景的需求：

  1. 本地 Docker 部署 - 用于本地开发和调试
  2. 远程（Server）Docker 部署 - 用于生产环境部署
  3. GitHub Pages 自动化部署 - 用于静态网站托管

🏗️ 三种部署方式

1. 本地 Docker 部署

用途: 本地开发和调试
特点: 热重载、实时预览、开发调试
命令: ./hexo.sh dev

优势:

  • ✅ 实时文件监控和热重载
  • ✅ 源代码挂载，修改即时生效
  • ✅ 完整的开发调试环境
  • ✅ 无需安装 Node.js 环境

2. 远程（Server）Docker 部署

用途: 生产环境部署
特点: 高性能、稳定可靠、生产就绪
命令: ./hexo.sh build && ./hexo.sh run

优势:

  • ✅ Nginx 高性能静态文件服务
  • ✅ Gzip 压缩和缓存优化
  • ✅ 健康检查和监控
  • ✅ 容器化部署，易于管理

3. GitHub Pages 自动化部署

用途: 静态网站托管
特点: 自动化、免费、CDN 加速
触发: 推送到 main 分支

优势:

  • ✅ 完全自动化部署
  • ✅ 免费静态网站托管
  • ✅ 全球 CDN 加速
  • ✅ 与 GitHub 生态集成

🌐 GitHub Pages 部署

双仓库架构

本项目采用双仓库架构，这是 Hexo 博客部署的最佳实践：

仓库说明

  1. 源码仓库 (zhifu-tech/zhifu-tech)

    • 存放 Hexo 博客的源代码
    • 包含 _posts/、themes/、_config.yml 等
    • 用于版本控制和协作开发
    • 触发 GitHub Actions 自动部署

  2. 部署仓库 (zhifu-tech/zhifu-tech.github.io)

    • 存放 Hexo 生成的静态文件
    • 自动启用 GitHub Pages 服务
    • 通过 GitHub Actions 自动更新
    • 用户访问的网站地址

自动化部署流程

环境变量设置

在 源码仓库 (zhifu-tech/zhifu-tech) 中设置以下密钥：

  1. 进入源码仓库 Settings → Secrets and variables → Actions
  2. 添加以下密钥：
密钥名称 描述 示例值
GIT_EMAIL_BLOG Git 邮箱地址 your-email@example.com
GH_TOKEN_BLOG GitHub Personal Access Token ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

创建 GitHub Personal Access Token

  1. 登录 GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
  2. 点击 Generate new token → Generate new token (classic)
  3. 填写信息：
    • Note: GitHub Actions Blog Deployment
    • Expiration: 选择过期时间
    • Scopes: 至少选择 repo 和 workflow
  4. 复制生成的 token 并保存

🔧 hexo.sh 脚本使用

脚本功能

hexo.sh 是项目的唯一入口，提供以下功能：

命令 功能 说明
./hexo.sh build 构建 Docker 镜像 构建生产和开发镜像
./hexo.sh run 运行生产环境 启动 Nginx 服务器
./hexo.sh dev 运行开发环境 启动热重载开发服务器
./hexo.sh stop 停止所有容器 停止运行中的容器
./hexo.sh clean 清理资源 清理容器和镜像
./hexo.sh status 查看状态 显示容器和镜像状态
./hexo.sh logs 查看日志 显示容器运行日志
./hexo.sh help 显示帮助 显示使用说明

有时浏览器缓存可能导致页面显示异常或者无法看到最新内容更新。请尝试清除浏览器缓存后重新加载页面：

  • Mac：按下 Cmd + Shift + R

若问题仍未解决，可尝试更换浏览器或清理浏览器缓存后再次访问。

📊 性能优化

Docker 镜像优化

  • 使用多阶段构建减少镜像大小
  • 使用 Alpine Linux 基础镜像
  • 配置 npm 镜像源加速安装
  • 生产镜像大小: ~50MB
  • 开发镜像大小: ~240MB

运行时优化

  • 配置 Nginx 缓存策略
  • 启用 Gzip 压缩
  • 设置合适的 worker 进程数
  • 配置日志轮转

🔒 安全配置

容器安全

  • 使用非 root 用户运行
  • 设置适当的文件权限
  • 配置安全头
  • 限制容器资源使用

网络安全

  • 只暴露必要端口
  • 配置防火墙规则
  • 使用 HTTPS（生产环境）
  • 定期更新基础镜像
